# 模块一：计算机网络与 HTTP 协议

## 学什么

本模块目标是理解互联网通信的基础，特别是 HTTP 协议的工作原理，为接口测试打下坚实的理论基础。

**核心概念清单：**
- **TCP/IP 模型**：理解四层结构（应用层、传输层、网络层、链路层）及数据封装过程。
- **HTTP 请求/响应结构**：**请求行/状态行**、**请求头/响应头**、**请求体/响应体**。
- **HTTP 方法**：**GET**、**POST**、**PUT**、**DELETE**、**PATCH** 等语义与幂等性、安全性。
- **HTTP 状态码**：**2xx**（成功）、**3xx**（重定向）、**4xx**（客户端错误）、**5xx**（服务器错误）。
- **HTTPS 与 TLS/SSL**：加密、认证与完整性保护的基本原理。
- **Cookie 与 Session**：状态保持机制，理解其区别与应用场景。

## 去哪学

1.  **书籍**：
    - 《HTTP 权威指南》（David Gourley, Brian Totty）：被誉为“HTTP圣经”，全面深入。
    - 《图解 HTTP》（上野宣）：图文并茂，适合快速入门建立直观理解。
2.  **视频课程**：
    - B站“计算机网络微课堂”（湖科大教书匠）：系统讲解网络原理，生动易懂。
    - 极客时间《透视 HTTP 协议》（罗剑锋）：从协议到应用，内容前沿。
3.  **实践工具**：
    - 浏览器开发者工具（Network 面板）：观察真实请求与响应。
    - `cURL` 命令行工具：手动构造和发送 HTTP 请求的最佳实践工具。

## 学不懂怎么办

**困惑1：HTTP 是无状态的，那网站是如何记住我登录状态的？**
回答：HTTP 协议本身不记录状态，但可以通过 **Cookie** 和 **Session** 机制在客户端和服务器端协作实现状态保持。登录成功后，服务器生成一个唯一 Session ID 返回给浏览器（通常通过 `Set-Cookie` 头），浏览器后续请求会自动携带此 Cookie，服务器通过 ID 找到对应的 Session 数据，从而识别用户身份。

**困惑2：GET 和 POST 请求到底有什么区别？仅仅是参数位置不同吗？**
回答：区别远不止参数位置（URL vs Body）。核心在于**语义**：GET 用于获取资源，应是幂等的（多次请求结果相同）且安全的（不修改服务器数据）；POST 用于提交数据，可能创建新资源或触发变更。因此，浏览器对 GET 和 POST 的行为也不同（如缓存、书签、回退）。

## 如何检验

1.  使用浏览器开发者工具，分析任意一个网页加载过程中的至少 5 个 HTTP 请求，并说明其方法、状态码和主要头信息的作用。
2.  仅使用 `cURL` 命令，完成一次向公共测试 API（如 `httpbin.org/post`）发送带 JSON 体的 POST 请求，并成功获取响应。
3.  解释为什么在浏览器中按 F5 刷新和 Ctrl+F5 强制刷新时，Network 面板中某些资源的状态码可能不同（涉及缓存相关头信息）。

# 模块二：Web API 架构与设计

## 学什么

本模块目标是掌握主流 Web API 的设计风格与规范，理解如何与 API 进行交互。

**核心概念清单：**
- **RESTful API 设计原则**：**资源**（Resource）、**URI** 命名、**统一接口**、**无状态**、**表述性状态转移**。
- **API 端点与版本管理**：常见的版本控制策略（URI路径、请求头）。
- **请求与响应数据格式**：**JSON**（JavaScript Object Notation）的结构与序列化/反序列化。
- **认证与授权**：**API Key**、**Bearer Token**（JWT）、**OAuth 2.0** 流程的基本概念。（https://cloud.tencent.com/developer/article/2418791    oauth介绍）
- **GraphQL 简介**：与 REST 的对比，核心概念如**查询（Query）**、**变更（Mutation）**、**模式（Schema）**。

## 去哪学

1.  **书籍/文档**：
    - 《RESTful Web APIs》（Leonard Richardson, Mike Amundsen）：REST 设计的经典著作。
    - 阮一峰的网络日志：《RESTful API 设计指南》（博客文章，清晰易懂）。
2.  **视频课程**：
    - Udemy 课程 “REST APIs with Flask and Python”：通过实践学习 REST 设计。
    - GraphQL 官方入门教程（https://graphql.org/learn/）：交互式学习，直观有效。
3.  **实践资源**：
    - GitHub REST API v3 官方文档：学习优秀 API 设计实例。
    - `httpbin.org`：一个用于 HTTP 请求测试的公共服务，支持多种方法和认证。

## 学不懂怎么办

**困惑1：RESTful 强调“无状态”，但很多 API 又需要 Token 认证，这不矛盾吗？**
回答：不矛盾。“无状态”是指**服务器不保存客户端的一次会话状态**。Token（如 JWT）本身包含了认证信息（如用户ID），客户端每次请求都携带它。服务器只需验证 Token 的有效性即可，无需在服务端保存会话信息。状态信息被转移到了 Token 这个“表述”中，由客户端负责携带。

**困惑2：GraphQL 声称可以“精确获取所需数据”，它如何解决 REST API 的“过度获取”问题？**
回答：在 REST 中，一个端点返回的数据结构是固定的。例如，请求 `/user/123` 可能返回用户的所有信息（包括不需要的字段）。而在 GraphQL 中，客户端需要显式地在查询中指定所需的字段。例如，查询 `{ user(id:123) { name, email } }` 只会返回 `name` 和 `email` 字段，避免了网络传输和服务端处理不必要的数据。

## 如何检验

1.  为一个简单的“博客系统”设计一套符合 RESTful 风格的 API，至少包含对文章（Post）和评论（Comment）资源的 CRUD 操作，并写出对应的 URI 和 HTTP 方法。
2.  使用 Postman 或 cURL，调用一个需要 Bearer Token 认证的真实 API（如 GitHub API 获取用户信息），并成功获得响应。
3.  对比说明在获取“用户及其最近3篇文章标题”这个需求下，RESTful API 和 GraphQL 分别如何请求，并分析其优劣。

# 模块三：接口测试核心方法论

## 学什么

本模块目标是建立系统的接口测试思维，明确测试范围、类型和流程。

**核心概念清单：**
- **接口测试范畴**：测试**接口**本身，而非前端界面，关注**数据交换、传递和控制管理**过程。
- **测试流程**：需求分析 → 测试设计（用例） → 环境准备 → 执行 → 报告与跟踪。
- **功能测试**：验证接口能否按照**设计文档**或**契约**（如 OpenAPI Spec）正确工作。
- **数据测试**：**边界值分析**、**等价类划分**、测试数据的**有效性**、**无效性**、**一致性**。
- **异常测试**：模拟**网络异常**、**参数异常**、**数据异常**、**依赖服务异常**等情况。
- **测试用例设计**：基于**参数**、**业务逻辑**、**状态迁移**来设计用例。

## 去哪学

1.  **书籍**：
    - 《软件测试》（Ron Patton）：经典入门，建立测试基础思维。
    - 《接口测试方法论与实践》（部分国内专家博客或社区文章合集）。
2.  **在线文档/博客**：
    - Martin Fowler 博客关于“测试金字塔”和“契约测试”的文章。
    - Postman 官方学习中心关于 API 测试策略的教程。
3.  **标准与工具**：
    - **OpenAPI Specification (Swagger)**：学习如何阅读和基于 API 文档设计测试用例。
    - **pytest** 官方文档：了解测试框架如何组织用例和断言。

## 学不懂怎么办

**困惑1：接口测试和 UI 自动化测试有什么区别和联系？**
回答：**区别**：接口测试在**协议层**进行，速度快、稳定性高、更早发现问题。UI 测试在**交互层**进行，速度慢、易受界面变化影响，但能验证端到端用户体验。**联系**：它们位于“测试金字塔”的不同层级。底层（单元、接口）的测试应该更多、更稳定，为顶层的 UI 测试提供质量保障。很多 UI 测试的问题根源在于接口。

**困惑2：如何对返回大量数据的列表查询接口进行有效测试？**
回答：核心是**分而治之**。1. **功能正确性**：验证分页参数（page, size）是否生效，总条数是否正确。2. **数据准确性**：验证排序（sort by）、筛选（filter）条件是否有效。3. **性能与稳定性**：请求超大页码或 size 时，接口是否优雅处理（如返回错误或默认限制）。4. **一致性**：确保列表中的单项数据与通过ID获取该单项详情的接口数据一致。

## 如何检验

1.  给定一个用户登录接口的文档（包含请求参数和成功/失败响应示例），设计出覆盖功能、数据和异常场景的至少 10 条测试用例。
2.  阐述“测试金字塔”模型，并解释为什么在自动化测试策略中应优先投资接口自动化测试而非 UI 自动化测试。
3.  针对一个“修改订单状态”的接口，分析其可能的状态迁移路径，并据此设计测试用例。

# 模块四：接口测试工具实践

## 学什么

本模块目标是熟练掌握手工接口测试的核心工具，提升测试效率和规范性。

**核心概念清单：**
- **Postman 核心功能**：**集合（Collection）**、**环境变量（Environment）**、**脚本（Pre-request Script, Tests）**、**变量**、**工作流**。
- **Postman 测试脚本**：使用 **pm.response**、**pm.environment** 等对象进行断言和环境管理。
- **cURL 命令详解**：各种参数（`-X`, `-H`, `-d`, `-b`, `--data-binary`, `-k`）的使用场景。
- **API 文档与测试**：导入 **OpenAPI (Swagger)** 文档生成测试集合。
- **Mock Server**：使用 Postman 或其他工具快速创建模拟接口，用于前后端并行开发或测试。

## 去哪学

1.  **官方资源**：
    - Postman 官方学习中心（Learning Center）：交互式教程，从入门到进阶。
    - cURL 官方手册（`man curl` 或 https://curl.se/docs/）：最权威的命令参考。
2.  **视频课程**：
    - B站/YouTube 搜索“Postman 接口测试全栈”：大量实战演示视频。
    - Udemy “Postman: The Complete Guide - REST API Testing”。
3.  **实践仓库**：
    - Postman 官方示例集合（如“Public Workspaces”中的“Postman API”）。
    - 利用 `httpbin.org` 作为练习目标，尝试各种 cURL 命令。

## 学不懂怎么办

**困惑1：Postman 的环境变量、集合变量、全局变量有什么区别，优先级如何？**
回答：这是作用域的区别。**全局变量（Globals）**：对所有集合生效，作用域最广。**集合变量（Collection Variables）**：仅对所在集合生效。**环境变量（Environment Variables）**：需要手动激活某个环境后才生效。**局部变量（Local）**：仅在单次请求的脚本中临时存在。**优先级**：局部 > 环境 > 集合 > 全局。当变量名冲突时，优先级高的生效。

**困惑2：在 Postman 的 Tests 标签页里写 JavaScript 断言，如何测试响应时间或处理复杂的 JSON 响应？**
回答：1. **响应时间**：使用 `pm.response.responseTime` 获取毫秒数，然后与预期值比较，如 `pm.expect(pm.response.responseTime).to.be.below(200)`。2. **复杂 JSON**：使用 `pm.response.json()` 将响应体解析为 JS 对象，然后利用 `pm.expect()` 的链式语法进行深度断言，例如 `pm.expect(jsonData.user.address.city).to.eql(‘Beijing’)`。也可以使用 `tv4` 库进行 JSON Schema 验证。

## 如何检验

1.  在 Postman 中创建一个集合，包含至少3个有依赖关系的请求（如：登录→获取Token→用Token访问受保护接口），并使用环境变量在请求间传递 Token。
2.  为上述集合中的一个 GET 请求编写测试脚本，断言其状态码为200，响应时间小于500ms，并且响应 JSON 中的某个数组长度大于0。
3.  不使用 Postman GUI，仅通过 cURL 命令完成一次包含文件上传（`multipart/form-data`）的接口请求。

# 模块五：接口自动化测试框架

## 学什么

本模块目标是将接口测试自动化，集成到开发流程中，实现持续测试。

**核心概念清单：**
- **Python `requests` 库**：`get()`, `post()`, `put()`, `delete()` 方法，处理**参数**、**头信息**、**Cookies**、**会话**。
- **测试框架 `pytest`**：**用例发现规则**、**夹具（fixture）**、**参数化（parametrize）**、**断言**、**钩子函数**。
- **测试数据管理**：数据与代码分离，使用 **YAML/JSON/Excel** 或 `@pytest.mark.parametrize`。
- **测试报告**：使用 `pytest-html`、`allure-pytest` 生成美观的测试报告。
- **持续集成**：将自动化测试脚本接入 **Jenkins**、**GitLab CI** 或 **GitHub Actions** 的基本思路。

## 去哪学

1.  **书籍**：
    - 《Python 接口自动化测试》（部分国内作者）：实战性强，贴合国内技术栈。
    - 《Python Testing with pytest》（Brian Okken）：深入理解 pytest 框架。
2.  **官方文档**：
    - `requests` 库官方文档（https://docs.python-requests.org/）：简洁明了，案例丰富。
    - `pytest` 官方文档（https://docs.pytest.org/）：权威指南，必读。
3.  **GitHub 仓库**：
    - `psf/requests`：阅读源码，理解高级用法。
    - 搜索 `api-testing-framework-python` 等关键词，参考开源项目结构。

## 学不懂怎么办

**困惑1：`pytest` 的 `fixture` 和 `setup/teardown` 有什么区别，什么时候该用哪个？**
回答：`setup/teardown`（如 `setup_module`）是 xUnit 风格的固定钩子，作用域固定（模块、类、方法）。`fixture` 更灵活强大：1. **可复用**：一个 `fixture` 可以被多个测试函数使用。2. **作用域可配置**：`function`（默认）、`class`、`module`、`session`。3. **依赖注入**：测试函数通过参数声明它需要的 `fixture`，`pytest` 自动注入。对于管理测试资源（如数据库连接、临时文件），优先使用 `fixture`。

**困惑2：如何优雅地处理测试用例中的动态数据，比如每次测试都需要一个新的用户名？**
回答：有多种策略：1. **使用固定前缀+随机数/时间戳**：在 `fixture` 或测试函数中生成，如 `f”test_user_{time.time()}”`。2. **预置数据池与清理**：在 `session` 级别的 `fixture` 中初始化一批数据，测试时取用，测试结束后统一清理。3. **依赖接口能力**：如果系统支持，通过接口先创建数据，返回其ID用于后续测试，最后再清理。核心原则是保证测试的独立
