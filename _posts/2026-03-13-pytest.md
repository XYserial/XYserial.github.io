# 模块一：Python基础与环境准备

## 知识点与讲解
本模块旨在为学习pytest自动化测试框架打下坚实的Python语言基础，并完成必要的开发环境配置。pytest本身是一个Python库，因此熟练掌握Python是高效使用pytest的前提。

### Python核心语法与数据结构
- **变量与数据类型**：理解Python的动态类型特性，掌握整数（`int`）、浮点数（`float`）、字符串（`str`）、布尔值（`bool`）等基本类型，以及`None`类型。
- **数据结构**：熟练掌握Python内置的四种核心数据结构：
    1.  **列表（List）**：有序、可变序列，用方括号`[]`表示。支持索引、切片、增删改查。
    2.  **元组（Tuple）**：有序、不可变序列，用圆括号`()`表示。常用于保证数据不被修改。
    3.  **字典（Dict）**：无序的键值对集合，用花括号`{}`表示，键必须是不可变类型。
    4.  **集合（Set）**：无序、不重复元素的集合，用花括号`{}`表示（但内部无键值对）。
- **流程控制**：掌握`if`/`elif`/`else`条件判断，`for`循环和`while`循环，以及`break`、`continue`、`pass`语句。
- **函数定义**：使用`def`关键字定义函数，理解参数（位置参数、默认参数、可变参数`*args`、关键字参数`**kwargs`）、返回值（`return`语句）和作用域的概念。

### 面向对象编程（OOP）基础
自动化测试中常用类来组织测试用例和测试数据。
- **类与对象**：类是对象的蓝图，对象是类的实例。使用`class`关键字定义类。
- **方法与属性**：类中定义的函数称为方法，第一个参数通常是`self`，代表实例本身。属性是绑定到对象的变量。
- **构造方法**：`__init__`方法在创建对象时自动调用，用于初始化对象状态。
- **继承**：允许一个类（子类）继承另一个类（父类）的属性和方法，实现代码复用和扩展。语法为`class ChildClass(ParentClass):`。

### Python模块与包管理
- **模块**：一个`.py`文件就是一个模块，可以通过`import`语句导入其功能。
- **包**：包含`__init__.py`文件的目录，用于组织多个模块。
- **pip**：Python的包管理工具，用于安装、卸载和管理第三方库。
    ```bash
    # 安装包
    pip install package_name
    # 安装指定版本
    pip install package_name==1.0.0
    # 从requirements.txt安装
    pip install -r requirements.txt
    ```

### 虚拟环境管理
为避免项目间依赖冲突，必须使用虚拟环境。
- **venv**：Python 3.3+ 自带的虚拟环境模块。
    ```bash
    # 创建虚拟环境（在当前目录下生成`venv`文件夹）
    python -m venv venv
    # 激活虚拟环境 (Windows)
    venv\Scripts\activate
    # 激活虚拟环境 (macOS/Linux)
    source venv/bin/activate
    # 停用虚拟环境
    deactivate
    ```
- **virtualenv**：第三方虚拟环境工具，功能更强大，兼容Python 2和3。

### 开发环境配置

## 题目与答案
**题目1**：请解释Python中列表（List）和元组（Tuple）的主要区别，并说明在自动化测试中，哪种数据结构更适合用来存储一组固定的测试数据（如测试用例的输入参数组合），为什么？

**答案**：列表和元组的主要区别在于可变性：列表是可变的（创建后可以修改、添加、删除元素），而元组是不可变的。在自动化测试中，**元组**更适合存储一组固定的测试数据。因为测试数据在测试执行过程中通常不应被意外修改，使用元组可以保证数据的完整性和一致性，避免因数据被篡改而导致的测试结果异常，这体现了防御性编程的思想。

**题目2**：请编写一个简单的Python函数 `calculate_statistics`，接收一个数字列表作为输入，返回该列表的最大值、最小值和平均值（保留两位小数）。然后，编写一个使用`assert`语句的代码片段来验证该函数对一个示例列表 `[1, 2, 3, 4, 5]` 的计算结果是否正确。

**答案**：
```python
def calculate_statistics(numbers):
    if not numbers:
        return None, None, None
    max_val = max(numbers)
    min_val = min(numbers)
    avg_val = sum(numbers) / len(numbers)
    return max_val, min_val, round(avg_val, 2)

# 使用assert进行验证
test_data = [1, 2, 3, 4, 5]
expected = (5, 1, 3.0)  # (max, min, avg)
result = calculate_statistics(test_data)
assert result == expected, f"Expected {expected}, but got {result}"
print("Test passed!")
```

# 模块二：软件测试与pytest入门

## 知识点与讲解
本模块将建立软件测试的基本概念，并引导你快速上手pytest框架，编写和运行第一个测试。

### 软件测试基础概念
- **测试目的**：验证软件行为是否符合预期，发现缺陷，提升质量。
- **测试金字塔**：一个经典的测试策略模型，强调应编写大量低层级的、快速的单元测试，适量集成测试，少量高层级的、慢速的端到端（UI）测试。
    | 测试类型 | 测试对象 | 速度 | 成本 | 隔离性 |
    | :--- | :--- | :--- | :--- | :--- |
    | **单元测试** | 单个函数/方法/类 | 快 | 低 | 高 |
    | **集成测试** | 模块/组件间交互 | 中 | 中 | 中 |
    | **端到端测试** | 完整系统/用户流程 | 慢 | 高 | 低 |
- **测试用例（Test Case）**：为特定目标而设计的一组输入、执行条件和预期结果。
- **断言（Assertion）**：测试的核心，用于验证实际结果是否等于预期结果。

### pytest框架简介与优势
pytest是一个成熟、功能丰富的Python测试框架。
- **主要优势**：
    1.  **简单易用**：使用简单的`assert`语句进行断言，无需学习额外的断言方法。
    2.  **自动发现**：自动发现以`test_`开头的文件、函数、方法。
    3.  **丰富的Fixture**：强大的测试夹具系统，用于管理测试前置和后置条件。
    4.  **插件生态**：拥有大量插件，可扩展功能（如测试报告、并发执行、数据库支持等）。
    5.  **详细输出**：测试失败时提供丰富的上下文和差异对比信息。

### 第一个pytest测试
1.  **创建测试文件**：文件名应以`test_`开头，如`test_calculator.py`。
2.  **编写测试函数**：函数名应以`test_`开头。
3.  **使用`assert`断言**：`assert`后跟一个布尔表达式，如果为`False`则测试失败。

**示例** (`test_basic.py`)：
```python
# 被测试函数
def add(a, b):
    return a + b

# 测试函数
def test_add_positive_numbers():
    assert add(2, 3) == 5

def test_add_negative_numbers():
    assert add(-1, -1) == -2

def test_add_mixed_numbers():
    assert add(5, -3) == 2
```

### 运行pytest测试
在命令行中，进入项目目录（确保虚拟环境已激活）：
- **运行所有测试**：`pytest`
- **运行特定文件**：`pytest test_basic.py`
- **运行特定函数**：`pytest test_basic.py::test_add_positive_numbers`
- **显示详细信息**：`pytest -v` （verbose模式）
- **遇到失败即停止**：`pytest -x`
- **常用选项组合**：`pytest -v -x`

### pytest断言详解
pytest使用Python原生的`assert`语句，但对其进行了增强，当断言失败时，会提供更易读的错误信息。
- **基本断言**：`assert expression`
- **断言预期异常**：使用`pytest.raises`上下文管理器。
    ```python
    import pytest

    def divide(a, b):
        if b == 0:
            raise ValueError("Cannot divide by zero!")
        return a / b

    def test_divide_by_zero():
        # 测试当除数为0时，是否抛出了ValueError异常
        with pytest.raises(ValueError) as exc_info:
            divide(10, 0)
        # 还可以进一步断言异常信息
        assert "Cannot divide by zero" in str(exc_info.value)
    ```

### 测试发现规则
pytest根据以下规则自动收集测试用例：
1.  在命令行指定的文件、目录中查找。
2.  递归查找子目录。
3.  匹配文件名：`test_*.py` 或 `*_test.py`。
4.  在匹配的文件中，收集：
    - 以`test_`开头的函数。
    - 以`Test`开头的类（且不能有`__init__`方法）中，以`test_`开头的方法。

## 题目与答案
**题目1**：请根据“测试金字塔”理论，比较单元测试、集成测试和端到端测试在测试对象、执行速度和维护成本上的主要区别。在自动化测试实践中，为什么建议将测试重点放在金字塔的底层？

**答案**：单元测试针对单个代码单元，执行速度极快，维护成本低；集成测试针对模块间接口，速度中等，成本中等；端到端测试模拟用户操作完整流程，速度慢，维护成本高。建议将重点放在金字塔底层（单元测试），是因为底层测试数量多、运行快、反馈及时、定位问题精准，能够以最小的成本获得最大的质量反馈，形成稳固的质量基座。上层的测试则作为必要补充，覆盖集成和用户场景。

**题目2**：请编写一个pytest测试文件 `test_string_utils.py`，其中包含一个被测试函数 `reverse_string(s)`，以及三个测试函数：1) 测试正常字符串反转；2) 测试空字符串反转；3) 测试回文字符串反转后与原字符串相同。

**答案**：
```python
# test_string_utils.py
def reverse_string(s):
    return s[::-1]

def test_reverse_normal():
    assert reverse_string("hello") == "olleh"

def test_reverse_empty():
    assert reverse_string("") == ""

def test_reverse_palindrome():
    test_str = "radar"
    assert reverse_string(test_str) == test_str
```

**题目3**：假设你运行`pytest`命令后，得到如下输出摘要：“FAILED test_calc.py::test_multiplication - assert 10 * 10 == 1000”。请分析这个失败信息，并解释pytest是如何增强原生的`assert`语句来提供更有帮助的失败信息的。

**答案**：该失败信息表明，在`test_calc.py`文件的`test_multiplication`函数中，断言`10 * 10 == 1000`失败。原生的Python `assert`只会抛出`AssertionError`，没有详细信息。pytest通过重写（rewriting）`assert`语句的字节码，在断言失败时能够捕获表达式的左右两部分（即`10*10`的结果`100`和字面量`1000`），并将它们清晰地展示在失败信息中，无需编写额外的断言消息，极大地便利了调试。

# 模块三：pytest核心功能详解

## 知识点与讲解
本模块深入探讨pytest框架的核心功能，这些功能是其区别于unittest等框架的关键，也是编写高效、简洁测试用例的基础。

### 测试发现与执行
pytest拥有强大的自动发现机制。它默认会递归查找当前目录及其子目录下所有以 `test_` 开头或 `_test` 结尾的Python文件，并执行其中所有以 `test_` 开头的函数和以 `Test` 开头的类中的以 `test_` 开头的方法。这是pytest“零配置”理念的体现。

### 断言机制
pytest使用Python原生的 `assert` 语句进行断言，这比unittest的 `self.assertEqual()` 等方法更简洁直观。当断言失败时，pytest会提供丰富的上下文信息，自动展示表达式的中间值，便于调试。
```python
def test_calculation():
    result = 5 * 6
    assert result == 30  # 断言成功
    assert result > 20   # 断言成功
    # 断言失败时，pytest会输出详细对比信息，例如：assert 30 == 25
```

### Fixture（夹具）
Fixture是pytest最强大、最核心的功能之一，用于为测试函数提供预置的上下文或数据。

#### 定义与使用
Fixture通过 `@pytest.fixture` 装饰器定义。测试函数可以通过将fixture的函数名作为参数来请求使用它。
```python
import pytest

@pytest.fixture
def sample_data():
    """提供一个简单的数据列表"""
    return [1, 2, 3, 4, 5]

def test_sum(sample_data):  # 通过参数名请求fixture
    total = sum(sample_data)
    assert total == 15
```

#### 作用域（Scope）
Fixture可以有不同的作用域，控制其创建和销毁的频率，优化测试性能。
- `function` (默认): 每个测试函数运行一次。
- `class`: 每个测试类运行一次。
- `module`: 每个模块（文件）运行一次。
- `package`: 每个包运行一次。
- `session`: 整个测试会话运行一次。
```python
@pytest.fixture(scope="module")
def shared_resource():
    # 模拟一个昂贵的初始化，如数据库连接
    conn = "Database Connection Established"
    yield conn  # 测试使用此连接
    # 测试结束后执行的清理代码
    print("\nClosing database connection.")
```

#### 自动使用（autouse）
将 `autouse=True` 的fixture无需在测试函数中显式请求，会自动应用于其作用域内的所有测试。
```python
@pytest.fixture(autouse=True, scope="session")
def global_setup_teardown():
    print("\n>>> Start Test Session")
    yield
    print("\n<<< End Test Session")
```

#### 常用内置Fixture
pytest提供了一些开箱即用的fixture：
- `tmp_path` / `tmp_path_factory`: 提供临时目录路径。
- `capsys`: 捕获标准输出和标准错误。
- `monkeypatch`: 动态修改对象、属性或环境变量。
```python
def test_output(capsys):
    print("Hello, pytest!")
    captured = capsys.readouterr()
    assert captured.out == "Hello, pytest!\n"
```

### 参数化测试
使用 `@pytest.mark.parametrize` 装饰器，可以用多组数据驱动同一个测试函数，避免编写重复代码。
```python
import pytest

@pytest.mark.parametrize("input_a, input_b, expected", [
    (1, 2, 3),
    (5, -5, 0),
    (3.5, 2.5, 6.0),
])
def test_addition(input_a, input_b, expected):
    assert input_a + input_b == expected
```
参数化也支持对测试类进行，为该类中所有测试方法提供数据。

### 标记（Mark）
标记用于对测试项（函数、类）进行元数据分类，以便选择性运行或特殊处理。
- **内置标记**:
    - `@pytest.mark.skip`: 无条件跳过测试。
    - `@pytest.mark.skipif`: 条件为真时跳过测试。
    - `@pytest.mark.xfail`: 预期测试会失败，不通过不影响整体结果。
- **自定义标记**: 在 `pytest.ini` 中注册后，可用于分组测试。
```python
@pytest.mark.slow
def test_expensive_operation():
    import time
    time.sleep(5)
    assert True

@pytest.mark.skipif(sys.version_info < (3, 8), reason="requires python3.8 or higher")
def test_feature_for_new_python():
    ...
```

### 插件体系
pytest的几乎所有功能都通过插件实现，其本身是一个小巧的核心。用户可以轻松安装和使用社区插件（如 `pytest-html` 生成报告， `pytest-cov` 集成覆盖率），或编写自己的插件来扩展框架功能。

## 题目与答案
**题目 1**：概念辨析题。请简要说明pytest的fixture与unittest框架中的setUp/tearDown方法的主要区别。
**答案**：主要区别在于灵活性和作用域。unittest的setUp/tearDown是固定的方法，作用于类或模块级别，所有测试共享相同的初始化和清理逻辑。而pytest的fixture是一个独立的、可声明的函数，具有明确的作用域（function, class, module, session），可以通过函数参数按需请求，支持依赖注入，并且可以通过yield实现更清晰的setup/teardown分离，灵活性远高于unittest。

**题目 2**：编码题。请编写一个测试函数，使用参数化测试多组登录场景（用户名、密码、预期结果），并使用 `pytest.mark.parametrize` 装饰器。
**答案**：
```python
import pytest

def login(username, password):
    # 这是一个模拟的登录函数
    if username == "admin" and password == "secret":
        return "登录成功"
    else:
        return "登录失败"

@pytest.mark.parametrize("username, password, expected", [
    ("admin", "secret", "登录成功"),
    ("admin", "wrong", "登录失败"),
    ("user", "secret", "登录失败"),
    ("", "", "登录失败"),
])
def test_login_scenarios(username, password, expected):
    result = login(username, password)
    assert result == expected
```

**题目 3**：分析题。假设你有一个需要连接数据库的测试，且这个连接在所有测试中只需要建立和关闭一次以提升效率。你会如何使用pytest fixture来实现？请描述fixture的scope应如何设置，并写出fixture的代码框架。
**答案**：为了实现数据库连接在整个测试会话中只建立和关闭一次，应该创建一个scope为 `"session"` 的fixture。这个fixture使用 `yield` 语句将数据库连接对象提供给测试函数，yield之后的代码会在所有测试结束后执行，用于关闭连接。代码框架如下：
```python
import pytest

@pytest.fixture(scope="session")
def db_connection():
    # 模拟建立数据库连接（实际中可能是 sqlalchemy.create_engine(...)）
    conn = {"status": "connected"}
    print("\n[Session] Database connected.")
    yield conn  # 将会话连接对象提供给所有测试
    # 所有测试结束后执行清理
    conn["status"] = "disconnected"
    print("\n[Session] Database disconnected.")
```

# 模块四：测试配置与高级执行

## 知识点与讲解
本模块聚焦于如何通过配置文件、命令行选项和插件来定制pytest的行为，以及如何高效地组织、筛选和执行测试用例。

### 配置文件
pytest可以通过配置文件来定义默认行为，优先级从高到低为：命令行选项 > 配置文件设置 > 默认值。

#### pytest.ini
这是最常用的配置文件，应放在项目根目录或测试目录的根目录。
```ini
[pytest]
# 指定测试文件搜索的目录
testpaths = tests unit_tests
# 指定测试文件名的模式
python_files = test_*.py check_*.py
# 指定测试函数/类的模式
python_functions = test_* check_*
python_classes = Test* Check*
# 自定义标记注册，防止拼写错误警告
markers =
    slow: marks tests as slow (deselect with '-m "not slow"')
    integration: integration tests
    smoke: smoke test suite
# 添加命令行默认选项
addopts = -v --tb=short
# 设置最低匹配的测试项数量，避免意外运行大量测试
minversion = 6.0
```

#### pyproject.toml
在遵循PEP 518的现代项目中，可以使用 `pyproject.toml` 来配置pytest。
```toml
[tool.pytest.ini_options]
testpaths = ["tests"]
markers = [
    "slow: marks tests as slow",
]
addopts = "-v"
```

#### conftest.py
这是一个特殊的文件，用于存放目录及其子目录共享的fixture、钩子函数和插件。pytest会自动发现并加载它。项目不同层级可以有自己的 `conftest.py`，fixture的作用域遵循就近原则。

### 命令行执行与控制
pytest提供了丰富的命令行选项来精确控制测试运行。

#### 选择运行哪些测试
- **按节点ID运行**: `pytest tests/test_module.py::TestClass::test_method`
- **按关键字表达式运行**: `pytest -k "login and not slow"` （匹配名称、标记、类名）
- **按标记运行**: `pytest -m slow` 或 `pytest -m "not integration"`
- **从上次失败处运行**: `pytest --lf` 或 `--last-failed`
- **运行指定文件/目录**: `pytest path/to/test_file.py`

#### 控制输出与回溯信息
- **详细输出**: `-v` (详细) / `-q` (安静)
- **回溯信息模式**: `--tb=style` (如 `short`, `no`, `long`(默认), `line`, `native`)
- **显示打印输出**: `-s` (禁用捕获) / `--capture=no`
- **显示fixture执行情况**: `--setup-show`

#### 测试执行与调试
- **收集但不运行**: `--collect-only`
- **在第一个失败后停止**: `-x` / `--exitfirst`
- **达到指定失败数后停止**: `--maxfail=2`
- **启动PDB调试器**: `--pdb` (在失败时) / `--trace` (在开始时)

### 测试分组与筛选策略
合理的测试分组是高效测试执行的关键。

| 分组维度 | 实现方式 | 适用场景 |
| :--- | :--- | :--- |
| **功能模块** | 目录/文件结构 | 大型项目，按业务模块划分 |
| **测试类型** | 自定义标记 (如 `@pytest.mark.unit`, `@pytest.mark.integration`) | 区分单元测试、集成测试、端到端测试 |
| **执行速度** | 自定义标记 (如 `@pytest.mark.slow`) | CI/CD中快速反馈，跳过耗时测试 |
| **优先级** | 自定义标记 (如 `@pytest.mark.smoke`, `@pytest.mark.p1`) | 冒烟测试、分级测试 |

### 插件管理与常用插件
pytest的强大生态体现在其丰富的插件上。
- **安装插件**: `pip install pytest-<plugin_name>`
- **列出已安装插件**: `pytest --version`

常用插件列表：
- **`pytest-html`**: 生成美观的HTML测试报告。
- **`pytest-cov`**: 集成覆盖率工具coverage.py，生成测试覆盖率报告。
- **`pytest-xdist`**: 实现测试的并行执行，显著缩短测试时间。
- **`pytest-ordering`**: 控制测试函数的执行顺序（慎用）。
- **`pytest-mock`**: 集成unittest.mock，提供 `mocker` fixture。
- **`pytest-asyncio`**: 对异步测试函数的支持。

### 钩子函数（Hooks）简介
钩子函数是pytest插件系统的基石，允许开发者在测试过程的特定节点插入自定义逻辑。它们定义在 `conftest.py` 或插件中。例如，`pytest_configure` 在配置阶段被调用，`pytest_collection_modifyitems` 可以在收集到所有测试项后修改它们（如重新排序、添加标记）。

## 题目与答案
**题目 1**：概念辨析题。请解释 `pytest.ini` 和 `conftest.py` 这两个文件在pytest项目中的主要作用和区别。
**答案**：`pytest.ini` 是一个**配置文件**，主要用于声明pytest的默认运行选项、自定义标记、测试文件命名规则等全局或目录级的设置，它不包含可执行代码。而 `conftest.py` 是一个**Python模块**，用于存放特定目录及其子目录共享的fixture函数、钩子函数和插件代码，pytest会自动导入它。前者配置行为，后者提供实现。

**题目 2**：动手题。假设你有一个测试套件，其中部分测试被标记为 `@pytest.mark.slow`。请写出两条不同的pytest命令：1) 仅运行这些慢测试；2) 运行除慢测试外的所有其他测试，并以简短格式 (`--tb=short`) 输出回溯信息。
**答案**：
1) 仅运行慢测试：`pytest -v -m slow`
2) 运行除慢测试外的所有测试，并使用简短回溯：`pytest -v -m "not slow" --tb=short`

**题目 3**：分析题。在一个持续集成（CI）环境中，你希望测试执行能更快地得到反馈。除了使用 `pytest-xdist` 进行并行测试外，还可以通过哪些pytest配置或命令行策略来优化测试执行时间？（请至少列出两点）
**答案**：在CI环境中优化测试执行时间，还可以采用以下策略：1) **选择性执行**：使用 `-m` 标记只运行冒烟测试（如 `-m smoke`）或关键路径测试，确保核心功能优先反馈。2) **失败优先与提前退出**：使用 `--lf` 先运行上次失败的测试，快速确认问题是否修复；结合 `-x` 或 `--maxfail=N` 在出现少数失败后立即停止，避免运行注定失败的后续测试，节省时间。

# 模块五：第三方集成与生态工具

## 知识点与讲解
pytest的强大不仅在于其核心功能，更在于其丰富的生态系统。本模块将系统学习如何将pytest与第三方工具集成，以构建更强大、更高效的测试工作流。

### 核心集成：Allure测试报告
Allure是一个轻量级、多语言支持的测试报告工具，能生成美观、交互式的测试报告。
- **定义与原理**：Allure通过监听测试执行过程，收集测试结果、步骤、附件、环境等信息，生成一个基于Web的HTML报告。pytest通过`pytest-allure`插件与其集成。
- **安装与配置**：
    ```bash
    pip install allure-pytest
    ```
- **基本使用**：
    1.  执行测试并生成结果文件：
        ```bash
        pytest --alluredir=./allure-results
        ```
    2.  生成并打开HTML报告：
        ```bash
        allure serve ./allure-results
        ```
- **装饰器与特性**：
    - `@allure.title(“测试用例标题”)`：自定义用例标题。
    - `@allure.story(“用户故事”)` / `@allure.feature(“功能模块”)`：用于报告中的分类。
    - `allure.attach(body, name, attachment_type, extension)`：在报告中附加文本、图片等。
    - `allure.step(“步骤描述”)`：定义测试步骤，使报告更清晰。
    ```python
    import allure

    @allure.feature("购物车")
    @allure.story("添加商品")
    def test_add_to_cart():
        with allure.step("打开首页"):
            # ... 操作代码
            pass
        with allure.step("搜索商品A"):
            # ... 操作代码
            allure.attach("搜索关键词：商品A", name="搜索日志", attachment_type=allure.attachment_type.TEXT)
        with allure.step("加入购物车"):
            # ... 操作代码
            assert True
    ```

### 测试覆盖率：pytest-cov
`pytest-cov`插件用于测量代码的测试覆盖率，帮助识别未被测试的代码区域。
- **安装**：`pip install pytest-cov`
- **常用命令**：
    - `pytest --cov=myproj tests/`：测量`myproj`包的覆盖率。
    - `pytest --cov=myproj --cov-report=html tests/`：生成HTML格式的覆盖率报告。
    - `pytest --cov=myproj --cov-report=term-missing tests/`：在终端输出报告，并显示未覆盖的行号。
- **配置**：可在`pyproject.toml`或`pytest.ini`中配置覆盖率选项。
    ```ini
    # pytest.ini 示例
    [tool:pytest]
    addopts = --cov=src --cov-report=html --cov-report=term
    ```

### 并行测试：pytest-xdist
`pytest-xdist`插件允许并行运行测试，显著缩短测试套件的执行时间。
- **安装**：`pip install pytest-xdist`
- **核心原理**：通过创建多个工作进程（worker），将测试集合分发给它们并行执行，最后汇总结果。
- **常用命令**：
    - `pytest -n auto`：自动检测CPU核心数并创建对应数量的工作进程。
    - `pytest -n 4`：指定使用4个工作进程。
    - `pytest --dist=loadscope`：按模块或类分组分发，确保同一组的测试在同一进程中运行，适用于有状态或依赖的测试。
- **注意事项**：
    - 并行时测试执行顺序是随机的。
    - 确保测试是独立的，不依赖共享状态或外部服务的特定顺序。
    - 对I/O密集型或网络请求密集型的测试提速效果更明显。

### Mock与Fixture增强：pytest-mock
`pytest-mock`提供了一个名为`mocker`的Fixture，它是对标准库`unittest.mock`的包装，使其在pytest中使用更便捷。
- **安装**：`pip install pytest-mock`
- **核心Fixture**：`mocker`
- **使用示例**：
    ```python
    # 假设我们有一个发送邮件的函数
    # src/email_sender.py
    def send_email(to, subject, body):
        # 复杂的SMTP调用
        pass

    # test_email.py
    def test_notify_user(mocker):
        # 使用 mocker.patch 替换真实函数
        mock_send = mocker.patch('src.email_sender.send_email')
        from src.notification import notify_user
        notify_user("user@example.com", "Hello")

        # 断言 mock 对象被以预期的参数调用
        mock_send.assert_called_once_with("user@example.com", "Notification", "Hello, user!")
    ```

### 参数化增强：pytest-lazy-fixture
`pytest-lazy-fixture`允许在`@pytest.mark.parametrize`装饰器中直接使用Fixture，极大地增强了参数化的灵活性。
- **安装**：`pip install pytest-lazy-fixture`
- **使用场景**：当参数化的值需要由复杂的Fixture生成时。
    ```python
    import pytest
    import pytest_lazyfixture as lf

    @pytest.fixture
    def first():
        return {'a': 1, 'b': 2}

    @pytest.fixture
    def second():
        return {'x': 10, 'y': 20}

    @pytest.mark.parametrize('data', [
        lf.lazy_fixture('first'),
        lf.lazy_fixture('second'),
    ])
    def test_process_dict(data):
        assert isinstance(data, dict)
        # 可以对 first 和 second 两个fixture返回的dict进行测试
    ```

## 题目与答案
**题目 1**：概念辨析题。请简述 `pytest-xdist` 插件中 `-n auto` 和 `--dist=loadscope` 这两个参数分别的作用和典型应用场景。

**答案**：`-n auto` 参数用于自动根据CPU核心数量创建对应的工作进程，以实现测试的并行执行，其典型场景是加速大型、独立的测试套件。`--dist=loadscope` 参数则用于控制测试分发策略，它确保同一个测试模块或测试类中的用例在同一个工作进程中运行，适用于测试用例间存在状态依赖或共享昂贵资源（如数据库连接）的场景。

**题目 2**：编码/动手题。请编写一个简单的测试函数 `test_upload_file`，使用 `allure` 装饰器将其标记为“文件管理”功能下的“上传”故事，并在测试步骤中模拟上传一个名为“report.pdf”的文件，最后在报告中附加一段文本“上传成功日志”。

**答案**：
```python
import allure

@allure.feature("文件管理")
@allure.story("上传")
def test_upload_file():
    with allure.step("选择文件 report.pdf"):
        # 模拟选择文件操作
        selected_file = "report.pdf"
        assert selected_file.endswith('.pdf')
    with allure.step("点击上传按钮"):
        # 模拟上传操作
        upload_success = True
        assert upload_success
        allure.attach("文件 report.pdf 上传成功，服务器返回状态码200。", 
                      name="上传成功日志", 
                      attachment_type=allure.attachment_type.TEXT)
```

**题目 3**：分析题。一个团队在使用 `pytest --cov` 生成覆盖率报告后，发现报告显示某关键函数 `calculate()` 的几行边界条件判断代码未被覆盖。请分析可能导致这个问题的两个原因，并分别提出解决方案。

**答案**：原因一：测试用例集未能构造出触发这些边界条件的输入数据。解决方案是补充针对边界值（如最小值、最大值、空值、非法输入）的测试用例。原因二：`calculate()` 函数可能被其他代码异常处理提前返回，或者这些边界条件代码本身是冗余的（死代码）。解决方案是首先审查 `calculate()` 函数的逻辑和调用上下文，确认这些代码是否真的可达，然后相应调整测试或生产代码。

### 分层测试策略
- **单元测试（Unit Tests）**：
    - **目标**：验证单个函数、方法或类的行为。
    - **位置**：`tests/unit/`
    - **特点**：快速、隔离（大量使用Mock）。
- **集成测试（Integration Tests）**：
    - **目标**：验证多个模块或与外部服务（数据库、API）的交互。
    - **位置**：`tests/integration/`
    - **特点**：速度较慢，需要准备外部环境。
- **端到端测试（E2E Tests，可选）**：
    - **目标**：模拟真实用户操作，验证整个应用流程。
    - **工具**：可与Selenium、Playwright等UI自动化工具结合。
    - **管理**：通常单独一个目录，如`tests/e2e/`。

## 题目与答案
**题目 1**：概念辨析题。请解释 `conftest.py` 文件中 Fixture 的 `scope` 参数设置为 `“function”`、`“class”`、`“module”` 和 `“session”` 时，其创建和销毁的生命周期有何不同。

**答案**：`scope=”function”`（默认）表示每个测试函数执行前后都会创建和销毁一次Fixture。`scope=”class”`表示在每个测试类中只创建一次，供该类所有测试方法使用。`scope=”module”`表示在每个测试模块中只创建一次。`scope=”session”`表示在整个pytest执行会话（即一次完整的测试运行）中只创建和销毁一次，是所有作用域中生命周期最长的，适合用于创建昂贵的资源如数据库连接。

**题目 2**：编码/动手题。请为上述实战项目中的 `src/app` 编写一个单元测试文件 `tests/unit/test_app_logic.py`。要求使用 `pytest-mock` 来模拟一个假设的 `email_validator` 模块（假设其有一个 `validate` 函数），测试 `create_user` 函数在邮箱验证失败时是否返回了正确的错误响应。

**答案**：
```python
# tests/unit/test_app_logic.py
import pytest
from unittest.mock import Mock
from src.app import app, users_db
from fastapi.testclient import TestClient

def test_create_user_with_invalid_email(mocker):
    # 模拟 email_validator.validate 函数返回 False
    mock_validate = mocker.patch('src.app.email_validator.validate', return_value=False)
    
    with TestClient(app) as client:
        response = client.post("/users/", json={"name": "Charlie", "email": "invalid-email"})
    
    # 断言验证函数被以正确的参数调用
    mock_validate.assert_called_once_with("invalid-email")
    # 断言API返回了错误（这里假设验证失败返回400）
    assert response.status_code == 400
    assert "Invalid email" in response.json()["detail"]
    # 断言用户没有被添加到数据库
    assert len(users_db) == 0
```
*(注意：此答案基于对原 `src/app.py` 逻辑的扩展，假设加入了邮箱验证和错误处理)*

**题目 3**：分析题。在一个大型微服务项目中，测试套件执行时间过长。团队计划引入 `pytest-xdist` 进行并行测试，但担心某些测试会因共享测试数据库而相互干扰。请结合 `conftest.py` 和 Fixture 作用域的知识，提出一个架构设计方案来解决数据库干扰问题。

**答案**：方案是使用 `pytest-xdist` 的 `--dist=loadscope` 参数，并结合会话级（`scope=”session”`）和函数级（`scope=”function”`）Fixture。首先，在会话级Fixture中为每个工作进程创建独立的数据库模式或连接（例如，通过为每个进程生成唯一的数据库名）。然后，在函数级Fixture中，使用该进程专用的连接，并在每个测试后回滚事务或清理该测试产生的数据。这样，不同进程的测试操作完全隔离的数据库，而同一进程内的测试通过事务隔离也能保证独立性。

# 模块七：源码分析与插件开发

## 知识点与讲解
本模块旨在深入pytest框架内部，理解其核心运行机制，并掌握基于此机制进行插件开发的能力。这是从框架使用者转变为贡献者和高级定制者的关键一步。

### pytest 核心架构
pytest的架构遵循“插件化”和“钩子（hook）”驱动的设计原则。其核心是一个轻量级的“内核”，所有功能（如测试收集、运行、报告）都由插件实现。主要组件包括：
- **`pytest` 主入口**：命令行调用触发整个流程。
- **配置对象（`pytest.Config`）**：存储命令行参数、配置信息，并在插件间共享。
- **插件管理器（`PytestPluginManager`）**：负责插件的加载、管理和钩子函数的调用。
- **钩子函数（Hook Functions）**：一系列预定义的函数接口，插件通过实现这些钩子来介入测试流程的特定阶段。

### 关键钩子函数详解
钩子函数是插件与pytest交互的桥梁。它们以 `pytest_` 开头，在 `conftest.py` 文件或插件模块中定义。

#### ### `pytest_addoption(parser)`
- **定义**：用于向pytest命令行添加自定义参数。
- **原理**：在初始化阶段被调用，接收一个 `argparse.ArgumentParser` 对象。插件通过 `parser.addoption` 方法添加参数。
- **示例**：
  ```python
  # conftest.py
  def pytest_addoption(parser):
      parser.addoption(
          "--env",
          action="store",
          default="test",
          help="指定测试环境：test/staging/prod"
      )
  ```
- **注意事项**：添加的参数值可以通过 `pytest.config` 对象（旧版）或 `request.config` 对象的 `getoption` 方法在测试中访问。

#### ### `pytest_collection_modifyitems(session, config, items)`
- **定义**：在测试用例收集完成后、执行前被调用，用于过滤或重新排序测试项。
- **原理**：接收 `items`（一个包含所有收集到的测试用例的列表），可以直接修改此列表。
- **示例**：根据标记（mark）过滤或按名称排序。
  ```python
  def pytest_collection_modifyitems(config, items):
      # 反转执行顺序
      items.reverse()
      # 只保留带有‘slow’标记的用例（危险操作，仅示例）
      # items[:] = [item for item in items if 'slow' in item.keywords]
  ```
- **注意事项**：`items` 列表中的每个元素都是一个 `Item` 对象（如 `Function`），代表一个测试用例。

#### ### `pytest_runtest_setup(item)` / `pytest_runtest_teardown(item)`
- **定义**：分别在每个测试用例执行的前置（setup）和后置（teardown）阶段被调用。
- **原理**：是运行测试的“夹具（fixture）”机制的一部分，但比 `@pytest.fixture` 更底层。
- **示例**：可用于在用例级别注入额外的资源管理。
  ```python
  def pytest_runtest_setup(item):
      print(f"准备运行测试：{item.name}")

  def pytest_runtest_teardown(item):
      print(f"测试结束清理：{item.name}")
  ```

### 插件开发流程
1.  **规划功能**：明确插件要解决的问题，确定需要介入的钩子阶段。
2.  **创建插件结构**：可以是一个独立的Python包（包含 `setup.py` 和 `hook` 实现模块），或一个项目内的 `conftest.py` 文件（用于本地插件）。
3.  **实现钩子函数**：在插件模块中编写具体的钩子函数实现。
4.  **注册与安装**：
    - **全局安装**：将插件打包，通过 `pip install` 安装，并在 `pytest.ini` 中通过 `addopts = -p myplugin` 或 `plugins = myplugin` 启用。
    - **本地使用**：将插件代码直接放在项目根目录或测试目录的 `conftest.py` 中，pytest会自动发现并加载。
5.  **测试与调试**：为插件本身编写测试，使用 `pytest` 运行以确保其行为符合预期。

### 常用内部对象与API
开发插件时，经常需要操作pytest的内部对象：
- **`request` 对象**：在测试上下文（如夹具函数内）可用，包含当前测试的配置、节点、模块等信息。
- **`Node` 及其子类**：`Session`, `Module`, `Class`, `Function`。它们构成了测试收集的树形结构，是 `items` 的具体类型。
- **`Mark` 对象**：处理测试标记，可通过 `item.keywords` 访问。
- **`FixtureDef`**：夹具定义，可通过 `request._fixturedefs` 查看。

### 调试与源码阅读技巧
- **使用 `--trace-config`**：运行 `pytest --trace-config` 可以查看所有已加载的插件和调用的钩子，是调试插件加载问题的利器。
- **设置断点**：在IDE中，直接在 `conftest.py` 或插件源码的钩子函数内设置断点进行调试。
- **阅读官方源码**：从 `pytest.main()` 入口开始，跟踪 `pytest_sessionstart` 等关键生命周期事件。

## 题目与答案
**题目1**：概念辨析题。请简述 `pytest` 的 `conftest.py` 文件与一个独立打包的 `pytest` 插件在功能、加载机制和使用范围上有何异同？

**答案**：
**相同点**：两者都是 `pytest` 插件的实现形式，都可以定义钩子函数和夹具来扩展或修改 `pytest` 的行为。
**不同点**：
1.  **功能与范围**：`conftest.py` 通常用于项目特定的、局部的定制，其夹具和钩子作用范围遵循目录层级规则。独立插件通常提供通用、可复用的功能，设计为一个Python包。
2.  **加载机制**：`conftest.py` 由 `pytest` 在测试收集过程中自动发现并加载，其加载顺序由目录结构决定。独立插件需要通过 `pip` 安装，并在配置文件（如 `pytest.ini`）中通过 `plugins` 列表显式注册，或使用 `-p` 命令行选项加载。
3.  **使用范围**：`conftest.py` 的作用域通常局限于当前项目或特定子目录。独立插件一旦安装并启用，可以对所有运行 `pytest` 的项目生效，更适合团队或社区共享。

**题目2**：编码/动手题。请编写一个简单的 `pytest` 插件（以 `conftest.py` 形式），实现以下功能：添加一个命令行选项 `--slow`，当指定该选项时，只运行那些被 `@pytest.mark.slow` 装饰的测试用例。

**答案**：
```python
# conftest.py
def pytest_addoption(parser):
    parser.addoption(
        "--slow",
        action="store_true",
        default=False,
        help="仅运行标记为 slow 的测试"
    )

def pytest_collection_modifyitems(config, items):
    if config.getoption("--slow"):
        # 保留带有 'slow' 标记的用例
        keep_items = []
        for item in items:
            if item.get_closest_marker("slow"):
                keep_items.append(item)
        # 原地修改 items 列表
        items[:] = keep_items
```
**解释**：`pytest_addoption` 钩子添加了 `--slow` 布尔选项。在 `pytest_collection_modifyitems` 钩子中，检查该选项是否被设置。如果设置了，就遍历所有收集到的测试项 (`items`)，使用 `item.get_closest_marker("slow")` 检查其是否带有 `slow` 标记，并将符合条件的项放入新列表，最后用新列表替换原 `items` 列表的内容。

**题目3**：分析题。假设你在开发一个插件时，发现你实现的 `pytest_runtest_protocol` 钩子函数没有被调用。请列出至少三种可能的原因及相应的排查步骤。

**答案**：
1.  **插件未正确加载**：使用 `pytest --trace-config` 命令查看输出，确认你的插件是否出现在已激活的插件列表中。如果没有，检查插件安装路径、`pytest.ini` 中的 `plugins` 配置或 `conftest.py` 文件的位置是否正确。
2.  **钩子函数签名错误**：检查 `pytest_runtest_protocol` 函数的参数签名是否与 `pytest` 官方文档一致。签名不匹配会导致钩子被静默忽略。正确的签名是 `pytest_runtest_protocol(item, nextitem)`。
3.  **与其他插件冲突**：可能存在另一个插件也实现了同一个钩子，并且其实现可能提前终止了钩子调用链。通过 `--trace-config` 查看钩子调用顺序，或临时禁用其他插件进行排查。此外，确保你的钩子函数返回了正确的值（如果需要），因为某些钩子的返回值会影响后续流程。
