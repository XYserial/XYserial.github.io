# 模块一：软件测试理论基础

## 学什么

软件测试是一切测试工作的地基。不管测什么（接口、性能、大模型），核心方法论是通用的。

**核心概念清单：**

- **测试类型**：功能测试 / 非功能测试 / 回归测试 / 冒烟测试（Smoke Test）/ 探索性测试
- **测试生命周期**：SDLC（软件开发生命周期）与 STLC（软件测试生命周期）的关系
- **用例设计方法**：
  - 等价类划分（有效类 + 无效类）
  - 边界值分析（刚好 / 刚好+1 / 刚好-1）
  - 判定表（多条件组合）
  - 正交法（减少用例数量）
  - 场景法（业务流程）
  - 错误推测法
- **缺陷管理**：Bug 生命周期、严重等级（P0-P4）、优先级、缺陷报告要素（标题/复现步骤/期望/实际/截图）
- **测试策略**：测试左移 vs 测试右移的概念（提前了解）

## 学不懂怎么办

**困惑1：等价类和边界值有什么区别，能合并用吗？**

回答：可以配合使用。等价类负责「划分范围」（比如年龄输入：有效类 1-120，无效类 <0 或 >120），边界值负责「找边界点」（0, 1, 120, 121）。两者互补，实际测试中先划等价类再取边界值是标准做法。

**困惑2：探索性测试是不是就是随便点点？**

回答：不是。探索性测试是有计划、有时间盒（Timebox）的测试活动，测试人员在执行过程中同步设计用例。区别在于它不依赖提前写好的用例，更依赖测试人员的经验和直觉，适合版本发布前的快速质量摸底。

**困惑3：Bug 严重等级和优先级有什么关系？**

回答：严重等级是技术影响（系统崩溃=P0，功能错误=P1，界面问题=P3），优先级是修复紧迫度（由业务决定）。一个界面错别字严重等级低，但如果在广告页面就优先级高。两者可以不一致。

## 如何检验

1. 拿一个真实 App（比如微信发红包功能），独立设计完整的测试用例集（覆盖正常流 + 异常流 + 边界）
2. 能口头解释 6 种用例设计方法的适用场景
3. 写一份规范的 Bug 报告（含复现步骤、截图、期望结果）
4. 能说清楚测试左移的好处和实施方式





### 检验一：拿微信发红包功能，设计完整测试用例集

**过关标准：** 独立写出 20 条以上有效用例，覆盖正常流、异常流、边界值，且每条用例要素完整（编号 / 标题 / 前置条件 / 步骤 / 期望结果）。

#### 操作方法：四步设计法

**第一步：梳理业务流程（场景法）**

先画出主流程和分支流程：

```
主流程：打开微信 → 进入聊天群 → 点「发红包」→ 填写金额/个数 → 确认支付 → 红包发出
分支流程：
  ├── 普通红包（固定金额）
  ├── 拼手气红包（随机金额）
  └── 专属红包（指定人）
异常流：余额不足 / 超出每日限额 / 网络中断 / 金额为0 / 金额超上限
```

**第二步：划等价类**

| 输入项 | 有效等价类 | 无效等价类 |
|---|---|---|
| 红包金额 | 0.01 ~ 200 元 | 0、负数、超过200、非数字 |
| 红包个数 | 1 ~ 100 个 | 0、负数、超过100、小数 |
| 祝福语 | 0 ~ 11 个字 | 超过11字、特殊字符（视业务） |
| 支付密码 | 6位数字 | 少于6位、含字母 |

**第三步：取边界值**

| 输入项 | 边界值测试点 |
|---|---|
| 红包金额 | 0（无效）/ 0.01（最小有效）/ 200（最大有效）/ 200.01（无效）|
| 红包个数 | 0（无效）/ 1（最小有效）/ 100（最大有效）/ 101（无效）|
| 祝福语 | 0字 / 1字 / 11字 / 12字 |

**第四步：补充异常场景**

- 发送过程中断网
- 余额不足时点击确认
- 支付密码输错3次
- 红包未被全部领取时退款
- 已过期红包（24小时后）退款验证

#### 用例表示例（节选5条）

| 编号 | 用例标题 | 前置条件 | 操作步骤 | 期望结果 |
|---|---|---|---|---|
| TC-001 | 正常发送拼手气红包 | 登录微信，余额≥0.01元 | 1.进入群聊 2.点「发红包」3.选拼手气 4.金额填1元 5.个数填3 6.确认支付 | 红包发送成功，群内显示红包消息 |
| TC-002 | 红包金额为0时无法发送 | 同上 | 填写金额0，点确认 | 提示「请输入有效金额」，无法提交 |
| TC-003 | 红包金额为0.01元（最小边界） | 同上 | 填写金额0.01，确认支付 | 发送成功 |
| TC-004 | 红包金额为200.01元（超上限） | 同上 | 填写金额200.01 | 提示「单个红包金额不超过200元」|
| TC-005 | 发送过程中断网 | 已填写金额，准备提交 | 提交前断开网络，点确认 | 提示网络错误，红包未扣款，可重试 |

### 检验二：口头解释 6 种用例设计方法的适用场景

**过关标准：** 对每种方法，能在 30 秒内说出「适合什么场景 + 不适合什么场景 + 一个具体例子」。

#### 速记对照表

| 方法 | 一句话定位 | 最适合的场景 | 不适合的场景 | 具体例子 |
|---|---|---|---|---|
| **等价类划分** | 「划范围，选代表」 | 输入字段有明确范围规则 | 无明显分类边界的功能 | 年龄输入框：有效类1-120，无效类<0或>120 |
| **边界值分析** | 「找临界，测极端」 | 数值/长度/日期有上下限 | 非数值类输入（如下拉选择） | 密码长度6-20位：测5、6、20、21位 |
| **判定表** | 「多条件，穷组合」 | 多个输入条件共同决定结果 | 条件超过5个（组合爆炸） | 登录：账号是否正确 × 密码是否正确 = 4种组合 |
| **正交法** | 「缩组合，保覆盖」 | 多参数需组合测试但数量太多 | 条件之间有强依赖关系 | 浏览器×操作系统×语言的兼容性测试 |
| **场景法** | 「走流程，覆分支」 | 业务有明确主流程+多分支 | 孤立功能点 | 电商下单：正常下单/余额不足/库存不足/超时未付 |
| **错误推测法** | 「凭经验，猜易错」 | 补充其他方法的遗漏 | 单独作为主要设计方法 | 上传文件：猜测"空文件""超大文件""特殊格式"会出错 |

#### 自测方式

对自己提问：「测试一个日期选择器，你会用哪几种方法？为什么？」

参考答案思路：

- 先用**等价类**划分有效日期范围（如 1900-01-01 到 2099-12-31）与无效范围
- 再用**边界值**取每月第1天、最后一天、2月28/29日（闰年边界）
- 用**错误推测法**猜测：2月30日、非法格式如「2024-13-01」会怎样
- 用**场景法**验证：跨月份选择、跨年选择、起止日期逻辑（开始日期不能晚于结束日期）

### 检验三：写一份规范的 Bug 报告

**过关标准：** 写出的 Bug 报告，让一个从未见过该问题的开发工程师，能在 5 分钟内独立复现。

#### Bug 报告完整模板

```
【Bug 标题】
格式：[模块] + [操作] + [异常现象]（简洁，见标题知问题）
示例：✅ 【发红包】金额输入200.01元时，未显示上限提示直接提交成功
     ❌ 发红包有问题（太模糊）

【环境信息】
- 设备：iPhone 15 Pro / Android 小米14
- 系统版本：iOS 17.3 / MIUI 14
- App 版本：微信 8.0.47
- 网络环境：4G / WiFi

【严重等级】
P0（崩溃/数据丢失）/ P1（核心功能不可用）/ P2（功能异常）/ P3（体验问题）/ P4（界面/文案）

【优先级】
高 / 中 / 低（结合业务影响判断）

【前置条件】
- 已登录账号，余额≥200.01元
- 处于任意聊天群中

【复现步骤】（编号列出，每步一个操作）
1. 打开微信，进入任意聊天群
2. 点击右下角「+」→ 选择「红包」
3. 选择「普通红包」
4. 在金额输入框中输入「200.01」
5. 点击「塞钱进红包」确认支付

【实际结果】
红包发送成功，用户被扣款200.01元，群内出现红包消息。

【期望结果】
金额超过200元时，应在输入阶段或点击确认时提示「单个红包金额不超过200元」，并阻止提交。

【附件】
- 截图1：输入200.01后的界面（未见任何提示）
- 截图2：红包发送成功的截图
- 视频：完整复现过程录屏（30秒）

【备注】
已测试200元可正常发送，201元也复现此问题。怀疑前端缺少上限校验。
```

#### 常见错误（对照自查）

| 错误写法 | 正确写法 |
|---|---|
| 标题：「红包金额有 bug」 | 标题：「发红包金额超上限200元时未弹出提示可直接提交」|
| 步骤混写：「打开红包然后输入金额然后点确认」 | 步骤编号：1.打开 2.输入 3.点击，每步独立 |
| 实际结果：「不对」 | 实际结果：具体描述看到了什么、发生了什么 |
| 缺少期望结果 | 必须写「应该发生什么」，让开发明白修复目标 |
| 无截图/视频 | 至少附1张截图，必要时录屏 |

### 检验四：说清楚测试左移的好处和实施方式

**过关标准：** 能用 2 分钟完整表达「是什么 → 为什么 → 怎么做 → 效果」，并能举出一个具体的落地例子。

#### 结构化答题框架

**① 是什么（定义）**

测试左移（Shift Left Testing）指把测试活动从传统的「开发完成后」提前到**需求阶段、设计阶段甚至开发阶段**介入，越早发现问题，修复成本越低。

「左」来自软件开发时间轴：需求 → 设计 → 开发 → **测试** → 上线，测试左移就是让测试向时间轴左侧移动。

**② 为什么（价值）**

缺陷修复成本模型（数据来自 IBM 系统科学研究）：

| 发现阶段 | 相对修复成本 |
|---|---|
| 需求阶段发现 | 1x |
| 设计阶段发现 | 5x |
| 开发阶段发现 | 10x |
| 测试阶段发现 | 20x |
| 上线后发现 | 100x |

越晚发现，修复越贵。测试左移的核心价值是**让廉价的早期修复取代昂贵的晚期修复**。

**③ 怎么做（实施方式，4 个具体动作）**

| 阶段 | 测试左移的具体动作 |
|---|---|
| 需求阶段 | 测试工程师参与需求评审，提前找出需求歧义和遗漏的异常场景（例：「用户转账失败后，钱去哪了？」） |
| 设计阶段 | 参与接口/架构评审，确认接口定义是否覆盖边界情况，提前编写测试用例草稿 |
| 开发阶段 | 推动开发做单元测试（Unit Test），测试工程师搭建 CI 流水线，代码提交即触发自动化测试 |
| 提测前 | 设置「准入标准」（Definition of Ready）：核心功能冒烟测试通过才能提测，不通过直接打回 |

**④ 效果（量化收益）**

- 需求缺陷占所有缺陷的 40-60%（来源：Capers Jones 研究），左移可大幅减少这类缺陷流到测试阶段
- 减少「提测 → 打回 → 返工」的循环浪费
- 测试工程师从「质量检验员」转变为「质量内建推动者」，职业价值更高

#### 一个具体落地例子

> **场景：** 电商平台上线「限时秒杀」功能，传统流程测试阶段才发现「库存扣减并发有问题」，已经延期2天。
>
> **测试左移做法：**
> 1. **需求评审时**，测试工程师提问：「100人同时抢最后1件商品，库存怎么保证不超卖？」→ 提前明确需要分布式锁方案
> 2. **接口设计评审时**，核对库存接口的并发设计，提前写好并发测试用例
> 3. **开发阶段**，CI 流水线中加入接口自动化测试，开发每次提交即验证库存逻辑
> 4. **结果：** 并发问题在开发阶段被单元测试发现，未进入测试阶段，节省3天修复+回归成本
